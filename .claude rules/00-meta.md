# 00. Мета — как пользоваться правилами

## Где живут правила

`C:\Users\dsadmin\Desktop\autocamera\.claude rules\` — единственный источник
правил по этому проекту. Папка лежит **в репозитории**, не в `.gitignore`
(в отличие от `.claude/`), поэтому правила версионируются вместе с кодом.

Каждый файл — отдельное правило. Имя начинается с двузначного номера, чтобы
порядок чтения был стабильным:

- `00-meta.md` — это правило
- `01-documentation.md` — обновление документации
- `02-git.md` — Git-workflow и что коммитить
- `03-secrets.md` — секреты и `.env`
- `04-style.md` — язык комментариев, стиль кода и логов
- `README.md` — индекс папки

## Когда Claude должен читать правила

- В начале каждой новой сессии — прочитать `README.md` (индекс) и весь
  `00-meta.md`.
- Перед действиями, которых касается конкретное правило, — открыть его
  целиком. Не угадывать содержание по названию.
- Если правила и `CLAUDE.md` противоречат — приоритет у `.claude rules/`
  (это более свежий и узкий источник). Сразу же предупредить пользователя
  и предложить синхронизировать `CLAUDE.md`.

## Формат файлов правил

Markdown с YAML-frontmatter в начале (как в примере Cursor):

```markdown
---
description: одна строка — о чём правило
globs:                       # пусто = применяется всегда
alwaysApply: true            # true = подгружать в каждый запрос
---

# NN. Заголовок

## Раздел
- пункт
- пункт
```

Поля:
- `description` — одна короткая строка, по ней правило ищется.
- `globs` — пути файлов, при изменении которых правило особенно важно
  (пусто = всегда).
- `alwaysApply: true` — означает «правило обязательно к учёту в любой задаче».

## Самодополнение правил (важно)

**Claude обязан сам расширять эту папку.** Если в ходе работы пользователь:

1. Корректирует подход («не делай так», «делай вот так», «всегда X»);
2. Подтверждает неочевидное решение («да, такой коммит правильно», «именно
   так и оформляй»);
3. Раскрывает контекст, который нужен будет в будущих сессиях (рабочий
   процесс, ограничения, договорённости с заказчиком);

— **сразу** оформить это как новое правило или дополнить существующее:

1. Найти подходящий файл по теме. Если нет — создать новый: `NN-тема.md`,
   где `NN` — следующий свободный номер.
2. Добавить запись в `README.md`-индекс (одна строка-ссылка).
3. Не дублировать то, что уже есть в `CLAUDE.md`, — лучше сослаться.
4. Писать **«почему»** вместе с **«что»**: одно правило без причины
   быстро устаревает.

После любого изменения правил — **сказать пользователю одной строкой,
что добавлено/изменено**, чтобы он мог отменить.

## Поддержание актуальности

- При каждом крупном изменении проекта (новая система камер, новый чекер,
  смена транспорта, новый внешний сервис) — пройтись по правилам и убрать
  то, что устарело.
- Не удалять историю молча: если правило сильно меняется, оставить под ним
  строку `> Обновлено YYYY-MM-DD: <причина>`.
- Хранение коротких заметок «на одну сессию» — это не правило. Для них есть
  task list и память (`C:\Users\dsadmin\.claude\projects\...\memory\`).

## Чего в правилах быть не должно

- Секретов, паролей, токенов, IP внутри VPN, путей с именами пользователей
  заказчика.
- Длинных кусков кода — для них есть сам код.
- Описания того, что уже исчерпывающе написано в `CLAUDE.md` или `README.md`.
  Тут только **рабочие соглашения**: как вести себя, что не забыть, как
  оформить, чего избегать.
