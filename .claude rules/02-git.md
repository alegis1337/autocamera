# 02. Git — что и как заливать

Команды «давай зальём на гит», «закоммить», «push» — это **полный цикл**:
проверить статус → решить, что в коммит, → обновить документацию → собрать
понятное сообщение → закоммитить → пушнуть (только если пользователь явно
сказал). Не дёргать `git add -A` и не пушить молча.

## Жёсткие запреты (никогда без явной команды пользователя)

- `git push --force`, `git push -f` — на любую ветку.
- `git reset --hard`, `git checkout .`, `git clean -fd`.
- `git commit --amend` на уже отправленных коммитах.
- `git rebase -i` — интерактивный режим в Claude Code не работает.
- Любые правки `git config`.
- `--no-verify`, обход hooks.
- Коммит из старой копии `C:\Users\dsadmin\autocamera\` — работаем в
  `C:\Users\dsadmin\Desktop\autocamera\`.

## Что НИКОГДА не попадает в коммит

Перечислено в `.gitignore`, но проверять глазами **перед каждым `git add`**:

- `.env`, `.env.bak` — пароли камер, SMTP, Bitrix-webhook, RTSP-credentials.
  Если случайно изменилось — откатить и не коммитить.
- `state/` — `helpdesk-state.json`, `timeline-*.json` (динамика, зависит
  от рантайма, мусорит историю).
- `screenshots/` кроме `.gitkeep` — кадры с камер заказчика.
- `logs/*.log` — могут содержать IP внутренней сети и трассировки.
- `reports/*.html` — содержат скриншоты + email-адреса.
- `node_modules/`.
- Любые `*.bak`, `*.tmp`, `nul`, `Thumbs.db`, `Desktop.ini`.

Если видишь чужой файл в `git status`, который ты не создавал
(например, файл рантайма), — **не добавляй**, спроси пользователя.
Это может быть его незакоммиченная работа.

## Что должно попасть в коммит

- Изменённый код в `src/`, `config/`, `menu.ps1`, `setup-schedule.ps1`.
- Все документы из таблицы в [01-documentation.md](01-documentation.md),
  которые задеты изменением.
- Обновлённый `.env.example`, если добавлены/удалены env-переменные.
- Новые/обновлённые правила в `.claude rules/`.
- `package.json` + `package-lock.json` вместе (если менялись зависимости).
- `.gitignore`, если появился новый тип runtime-мусора.

## Стейджинг

- Добавлять файлы **по именам**: `git add src/foo.js CLAUDE.md`.
- Не использовать `git add -A` / `git add .` — слишком легко зацепить
  `state/` или новый файл с секретом.
- Перед коммитом обязательно: `git status` + `git diff --staged` —
  посмотреть глазами, что уходит.

## Формат сообщения коммита

```
<type>: <короткое описание на русском, повелительное наклонение>

<необязательное тело — что и зачем, переносы по 72 символа>
```

Типы:

- `feat` — новая функциональность (новая система камер, новый CLI-флаг,
  новый чекер).
- `fix` — исправление бага в существующем поведении.
- `refactor` — переписал код без изменения поведения.
- `perf` — улучшил производительность.
- `docs` — только документы (`CLAUDE.md`, `README.md`, `.env.example`,
  `.claude rules/`, `autocamera-manual.docx`).
- `chore` — зависимости, конфиги, скрипты установки.
- `style` — форматирование, переименование переменных.
- `test` — тесты.

Примеры:

```
feat: добавлен чекер TP-Link Tapo через RTSP+ffmpeg
fix: --only больше не сохраняет state и не шлёт helpdesk
docs: правила .claude rules — git и документация
```

Подпись `Co-Authored-By: Claude <noreply@anthropic.com>` (или с указанием
конкретной модели) в конце тела коммита **разрешена** — пользователь
согласился считать Claude соавтором правок, сделанных в сессии.
Обновлено 2026-05-27.

## Перед коммитом — чеклист

1. `git status` — что вообще изменилось.
2. Сверить с [01-documentation.md](01-documentation.md): все ли документы
   обновлены?
3. Проверить, что `.env` / `state/` / `screenshots/` / `logs/` / `reports/`
   не попали в staged.
4. `git diff --staged` — нет ли там паролей, IP внутренней сети, ключей
   Bitrix-webhook, email-адресов заказчика.
5. Собрать сообщение по формату выше.
6. Закоммитить.

## Push

- `git push` делаем **только если пользователь явно сказал** «запушь»,
  «отправь на гит», «push».
- На `main` пушим напрямую (одиночная разработка, без PR).
- После push — отчитаться одной строкой: что закоммичено, что запушено,
  какие документы тронуты.

## Откат

- Если коммит ещё не запушен и оказался плохим — `git reset --soft HEAD~1`
  (бережно, файлы остаются изменёнными) и пересобрать коммит.
- `git reset --hard` — только с явного разрешения, теряет несохранённое.
