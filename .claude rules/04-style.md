# 04. Стиль — язык и поведение

## Язык

- **Весь русский**: комментарии в коде, логи, сообщения коммитов,
  пользовательский текст в email/helpdesk, сообщения в чате
  с пользователем. (См. memory `feedback_russian_comments.md`.)
- Английский допустим только там, где иначе нельзя: ключи JSON, имена
  функций/переменных, типы коммитов (`feat:`, `fix:`), CLI-флаги.
- Никаких эмодзи в коде и документах, если пользователь явно не просит.
  В чате — тоже воздержаться.

## Стиль кода

- Node.js ≥ 18, **ES modules** (`import`/`export`, не `require`). Это
  заложено в `package.json` (`"type": "module"`).
- Не выдумывать абстракции «на будущее». Простой код, мало слоёв.
- Не плодить try/catch вокруг кода, который не может упасть. Ловить
  ошибки там, где есть план обработки, не «на всякий случай».
- Комментарии — только когда нужен **почему**, а не **что**. Что и так
  видно из кода.

## Поведение агента (рантайм)

Проект работает **без человека** — Task Scheduler запускает скрипт по
расписанию. Поэтому:

- Никаких `readline`, `prompt`, `inquirer`, ожиданий ввода.
- Никаких бесконечных циклов без `timeout`-ов.
- Любой `await` к внешнему ресурсу (камера, SMTP, SMB, RTSP) — с
  таймаутом. Камера может не ответить никогда.
- Отдельная упавшая камера / упавшая система — **залогировать и идти
  дальше**, не падать всем процессом. Падение оправдано только если
  упало что-то общее (Playwright не стартанул, SMTP недоступен в
  `--test-email`).
- Exit code: `0` — успех, `1` — фатальная ошибка (нужна для
  Task Scheduler-овых уведомлений).

## Логи (`src/logger.js`)

- Формат: `[ISO_TIMESTAMP] LEVEL текст`.
- Уровни: `INFO`, `WARN`, `ERROR`. Без `DEBUG` в проде — он за флагом
  `--debug`.
- Текст логов — на русском.
- В лог попадает: что начали, что закончили, сколько заняло, что
  обнаружили (камера офлайн / запись не свежая), ошибки с контекстом.
- В лог **не попадает**: пароли, токены, тело RTSP-URL с
  `user:pass@`, длинные HTTP-ответы целиком (только статус + размер).
- Файл: `logs/YYYY-MM-DD.log` (день = один файл).

## Поведение Claude в чате

- Короткие ответы. Если задача мелкая — одна-две фразы и сразу делаем.
- Перед каждым tool-call коротко сказать, что делаешь. Не молчать,
  но и не комментировать каждую мысль.
- В конце задачи — одна-две строки: что изменилось, что дальше.
- Не предлагать «давай ещё на всякий случай отрефакторим» — только то,
  что попросили.
- Уточняющие вопросы — только если действительно блок. В auto-режиме
  биас — делать разумный выбор и продолжать.
